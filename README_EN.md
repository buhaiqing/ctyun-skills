# Skills Farm - CTyun Skills Development Guide

This project is a **CTyun (天翼云) Operations Agent Skills collection** that provides
automated operations, monitoring, and management capabilities for cloud products.
Built on standard Markdown + [Agent Skills OpenSpec](https://agentskills.io/specification),
following the `ctyun`-first with SDK fallback strategy, and consumable by multiple
AI Agent runtimes (Harness, Claude Code, Cursor, etc.).

## Core Value

**Skills Farm is a Meta Skill system** — transforming operational knowledge into
structured, AI Agent-parseable, executable, and verifiable declarative specifications.

### Key Features

| Feature | Description |
|---------|-------------|
| **Placeholder Mechanism** | `{{env.*}}` (env vars), `{{user.*}}` (user input), `{{output.*}}` (output capture) — enables the human-machine dual channel |
| **Delegation** | `SHOULD/SHOULD NOT Use` defines boundaries; cross-product operations are auto-routed |
| **ctyun-first Execution** | CLI is the primary path; 3 retries with exponential backoff, then SDK/API fallback. See [CLI-First Decision Matrix](ctyun-skill-generator/references/cli-decision-matrix.md) |
| **"CLI 不满足" Classifier** | 4-class detection (Environment / Capability / Runtime / Business) + **per-operation** fallback granularity |
| **Generator-Critic-Loop (GCL)** | Adversarial quality gate: every skill has `rubric.md` + `prompt-templates.md` + trace persistence |
| **Token Efficiency** | Per-skill line budgets + cross-skill dedup table + "Reference, don't copy" style |
| **Doc Link Integrity** | `lychee` + placeholder regex + `## Quality Gate (GCL)` presence — 7-line Pre-merge Gate |
| **Skill Lifecycle & Doc Sync** | Planned / Shipped / Retired states + 6-row Sync Matrix + doc-drift anti-patterns |
| **Security** | Credential isolation (CLI reads only `~/.ctyun/config`; SDK reads `{{env.*}}`) + operation safety gates (delete/recovery requires explicit confirmation) |
| **Cross-platform** | Standard Markdown + OpenSpec; pluggable into multiple Agent frameworks |

### What is Meta Skill

A **Meta Skill** is a "Skill that generates Skills" — not specific cloud product
operations, but the **capability to produce operational knowledge**.

| Comparison | Regular Skill | Meta Skill |
|------------|---------------|------------|
| Responsibility | Execute specific ops tasks | Generate / orchestrate other Skills |
| Example | `ctyun-ecs-ops` manages VMs | `ctyun-skill-generator` generates new Skills |
| Input | Env vars + user instructions | Product doc URL + OpenAPI definition |
| Output | Execution results | Structured Skill documents |

> Core Principle: Credentials marked with `{{env.*}}` **MUST NEVER be requested from users**,
> preventing leakage at the mechanism level.

> **One-line summary**: Skills Farm enables AI Agents to evolve from "answering questions"
> to "autonomous operations" — every CTyun product gets an "AI-native" operations assistant.

## What is a Skill

A structured Markdown document that guides an AI Agent through a specific task.
Includes: YAML frontmatter metadata, Trigger & Scope, placeholder conventions,
Pre-flight → Execute → Validate → Recover flow, Output Parsing, Failure Recovery,
Safety Gates, and a `## Quality Gate (GCL)` section (per-skill GCL parameters).

## CTyun CLI

All Skills interact with CTyun through the official `ctyun` CLI. **CLI is primary**;
SDK/API is the contingency.

### Installation

**Method 1: `uv` (recommended, idempotent and reproducible)**

Use [uv](https://docs.astral.sh/uv/) for Python environment management:

```bash
# Install uv (one-time)
curl -LsSf https://astral.sh/uv/install.sh | sh

# Create env and install (idempotent, safe to re-run)
uv venv --python 3.10
source .venv/bin/activate  # Windows: .venv\Scripts\activate
uv pip install ctyun-cli ctyun-sdk
```

**Method 2: pip**

```bash
pip install ctyun-cli ctyun-sdk
```

**Method 3: project pyproject.toml**

```bash
uv sync
```

### Configure Credentials

> **Important:** the `ctyun` CLI **does NOT read environment variables** — it only reads
> `~/.ctyun/config` (INI format). SDK mode reads `{{env.*}}`.

#### Method 1: `.env` file (recommended for local dev, SDK mode)

```bash
cp .env.example .env
# Edit with real credentials
```

```ini
# .env example (for SDK mode)
CTYUN_ACCESS_KEY=your_access_key_here
CTYUN_SECRET_KEY=your_secret_key_here
CTYUN_REGION=cn-gz
```

#### Method 2: CLI config file (CLI mode)

```bash
# Interactive
ctyun config init

# Or manual
export HOME=/tmp/ctyun-home    # sandbox-safe writable HOME
mkdir -p /tmp/ctyun-home/.ctyun
cat > /tmp/ctyun-home/.ctyun/config << 'EOF'
[default]
access_key = YOUR_ACCESS_KEY
secret_key = YOUR_SECRET_KEY
region_id = cn-gz
endpoint = ecs.ctyun.cn
scheme = https
timeout = 20
EOF
printf "%s" "default" > /tmp/ctyun-home/.ctyun/current
```

> ⚠️ **The CLI config file does NOT accept `{{env.*}}` placeholders** — render to literal
> values with `envsubst` / `sed` before writing.

> ⚠️ **Security warning:** **NEVER** print `CTYUN_SECRET_KEY` value in logs, console output,
> or debug messages. When verifying credentials, only check existence (e.g.
> `test -n "$CTYUN_SECRET_KEY"`). For status logging, use masked placeholders like
> `CTYUN_SECRET_KEY=<masked>`.

#### Method 3: Shell env vars (**SDK mode only**)

```bash
export CTYUN_ACCESS_KEY="your_access_key"
export CTYUN_SECRET_KEY="your_secret_key"
export CTYUN_REGION="cn-gz"
```

> **Credential priority** (SDK mode): Shell env > `.env` > default. CLI mode only reads
> `~/.ctyun/config`.

## ctyun-first with SDK Fallback

Every `ctyun-*-ops` skill **MUST** follow this execution policy:

```
ctyun CLI exists? ── yes ──► try CLI (3 retries, exp backoff) ──► success? continue
                                                              └─ 3 fails ──► SDK fallback (per-op)
   └─ no / unknown ──► mark `cli_applicability: sdk-only` for this op; use SDK/API
```

The **full decision table** (9-item "CLI 不满足" classifier, 4-class fallback scope,
session-level policy, 5 anti-patterns) is in
[`ctyun-skill-generator/references/cli-decision-matrix.md`](ctyun-skill-generator/references/cli-decision-matrix.md).

The **repo-wide policy header** is in
[`AGENTS.md` §CLI-First Policy (Repository-Wide)](AGENTS.md#cli-first-policy-repository-wide).

> **Repository rule:** If a `ctyun <product> <op>` subcommand exists, use it. SDK/API is a
> contingency, not a parallel first choice.

### Hard-Earned ctyun CLI Quirks

| Quirk | Wrong | Correct |
|-------|-------|---------|
| `--output json` placement | `ctyun ecs list --output json` | `ctyun --output json ecs list` |
| `--no-interactive` flag | `ctyun --no-interactive ecs delete` | **Omit entirely** — not supported |
| Credentials | `export CTYUN_ACCESS_KEY=...` (CLI ignores) | Must write `~/.ctyun/config` INI file |
| `~/.ctyun/current` newline | `echo "default" > file` (adds newline) | `printf "%s" "default" > file` (no trailing newline) |

## Develop a New Skill

### 1. Reference the generator

Load these files as input context:

```
@ctyun-skill-generator/SKILL.md
@ctyun-skill-generator/references/ctyun-skill-template.md
@ctyun-skill-generator/references/cli-decision-matrix.md
@ctyun-skill-generator/references/rubric.md
```

### 2. Provide the prompt

> "Generate a CTyun [product] Skill, named `ctyun-xxx-ops`, core functions [description]. OpenAPI docs: [URL]"

### 3. Generated structure

```
ctyun-[product]-ops/
├── SKILL.md                              # MUST contain ## Quality Gate (GCL) section
├── assets/example-config.yaml
└── references/
    ├── cli-usage.md
    ├── core-concepts.md
    ├── integration.md
    ├── monitoring.md
    └── troubleshooting.md
```

### 4. Quality Bar (P0 — must pass)

- **Trigger & Scope** with SHOULD-use / SHOULD-NOT-use + delegation
- **Variables**: `{{env.*}}` / `{{user.*}}` / `{{output.*}}`; no real secrets
- **Flows**: Pre-flight → Execute → Validate → Recover; **every op must document both the
  ctyun path AND the SDK path inline**
- **CLI fidelity**: subcommands / flags match the official `ctyun` CLI docs; JSON paths
  **verified with a real `--output json` run**
- **Safety gates** for destructive ops (both ctyun path and SDK path)
- **CLI-First Decision Matrix reference**: declare
  `fallback_decision_table: references/cli-decision-matrix.md` in the
  `## Quality Gate (GCL)` section
- **Token Efficiency**: SKILL.md ≤ 600 lines; each `references/*.md` ≤ 800 lines
- **Doc Integrity**: every relative link resolves; `{{var.*}}` matches the
  `{{env.*}}` / `{{user.*}}` / `{{output.*}}` contract

## FAQ

| Q | A |
|---|---|
| Skill vs MCP Server? | A Skill is documentation; MCP is an execution service. |
| One Skill, many products? | Recommend single responsibility; cross-reference via `references/`. |
| What is ctyun-first? | The CLI is the primary path; SDK/API is only used when the CLI is **unsatisfied** (see [CLI-First Decision Matrix](ctyun-skill-generator/references/cli-decision-matrix.md)). |
| What is GCL? | Generator-Critic-Loop: an adversarial quality gate where every skill has a rubric and a Critic. See [AGENTS.md §GCL](AGENTS.md#generator-critic-loop-gcl--adversarial-quality-gate). |
| How is "CLI 不满足" detected? | 4 classes: Environment (fix & retry) / Capability (mark `sdk-only`) / Runtime (per-op fallback) / Business (never fallback). See [Decision Matrix §2](ctyun-skill-generator/references/cli-decision-matrix.md#2-what-counts-as-cli-不满足-cli-unsatisfied). |
